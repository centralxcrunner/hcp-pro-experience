Key Context

~2/3 of inbound CSRAI calls are from existing customers. The current prompt treats all callers the same — one mode for all contexts. The AI needs to detect intent at call start and switch between Service Mode (existing customer with active appt) vs Sales Mode (no active appt → book now).

📆 Availability Dead-Ends — What's Actually Happening

178 calls (10%) end with no booking because availability comes back empty. But "no availability" isn't one problem — it's at least three distinct failure modes, each with a different fix.

Type 1 — Pro Config Broken
OLB disabled or calendar not set up

fetch_availability returns zero because the pro's booking settings aren't configured — OLB disabled on services, no calendar availability, or service not bookable. The AI has no way to know why. The pro has no idea this is happening. Every incoming call for that service silently fails.

Bill's Appliance Repair OH · 3/9
CUSTOMERI need my refrigerator checked out.
AII'm not seeing any appointment openings in the system right now for refrigerator service, but we can still help.
CUSTOMERWhen would they be able to get out here? Tomorrow?
AII can't see any open slots showing in the system right now, but I'll note that you're hoping for tomorrow and have the team reach out.
→ Booking never confirmed. Pro likely doesn't know CSRAI failed this customer.
Fix: alert the pro when fetch returns zero. "CSRAI couldn't find availability for [service] — check your booking settings."
Type 2 — AI Offered a Time It Didn't Have
Placeholder time offered before fetch — then fetch fails

Flagged by Andre (Mar 2): AI offers "Monday morning" as a placeholder before calling fetch_availability. When fetch returns empty, the AI is stuck — it already set an expectation it can't fulfill. Customer may believe they have a booking. Also seen: AI offers days not marked available in pro's settings (Bianchi Winery — offered Thursday when Thursday wasn't bookable).

Columbia Electrical · 3/9
CUSTOMERNext Monday at nine AM.
AII checked next Monday and I'm not seeing any openings available yet. What's your preferred time window?
CUSTOMERJust ask Columbia to call me.
→ AI asked for preference before knowing if any slots exist. Customer lost confidence and disengaged.
Fix: never offer or ask for a time preference before fetch runs. Fetch first, then present real options. (Requires pre-loading slots at call start — Phase 2.)
Type 3 — Calendar Genuinely Full
Real no availability — but AI gives up instead of offering fallback

Sometimes the calendar is just full. That's legitimate. The problem is the AI treats it as a dead end — says "we don't have anything" and takes notes. It should always offer the OLB link as a fallback so the customer can self-book the moment a slot opens, and offer to note them for an earlier opening (waitlist behavior).

Texas Pride Heating & Air · 3/11
CUSTOMERTrying to make an appointment for tomorrow.
AII'm not seeing any available times at all for a tune up. What day would you like me to check next?
CUSTOMERThursday.
AII checked Thursday — not seeing any openings. Would you like me to check Friday?
CUSTOMERJesus.
→ Checked 4 days one at a time. Never offered next available across the week. Never offered OLB link.
Fix (prompt): always find first available across 7 days in one call. If nothing in 7 days, offer OLB link by text. Never check one day at a time.
🔔 Product Gap — Pro Never Notified When CSRAI Can't Book

Across all three types, the common thread is that the pro has no visibility into what CSRAI is failing to do. When fetch returns empty, no alert fires. When a booking API call fails, no flag is set. When a customer hangs up frustrated, there's no notification. The pro only finds out if they review the call log on desktop — which solo operators in the field almost never do in real time.

Needed: a pro-facing alert when CSRAI encounters zero availability — type, service, org, timestamp — so they can check settings or call the customer back.

🏠 Address Handling — Best Practices for New Customers

Address collection is the highest-friction task in voice AI. Two of the top four issues in this analysis (validation loop, address repetition) are address-related. The following recommendations come from industry research on voice AI best practices, combined with what we're seeing in the CSRAI data.

What CSRAI does today
  • Asks for full address in one turn: "What's the full address including city and zip?"
  • Fires validation immediately, narrates it ("Just a moment while I validate…")
  • Reads the validator's result back — even when wrong
  • Loops if customer disagrees (18% of calls)
  • Repeats the full address on every subsequent turn (29% of calls)
  • Never asks for unit/apartment separately
  • Asks for full address again even for known returning customers
What it should do
  • For known callers: confirm on-file address in one turn, skip collection entirely
  • For new customers: collect zip first (fast service area gate, narrows valid streets)
  • Then collect street number + street name as separate turns if needed
  • Ask for unit/apartment explicitly as a standalone question
  • Read back address digit-by-digit once to confirm, then stop
  • Validate silently — never narrate it, never fight the customer
  • Accept customer override immediately on first disagreement

6 Specific Recommendations

1. ZIP code first — always

Start with ZIP code, not full address. This (a) immediately gates service area — you know within seconds if you can serve them, (b) narrows the universe of valid street names for the validator, and (c) is much easier for a customer to say on a voice call than a full address.

Prompt copy
For new customers, collect address in this order:
1. "What zip code are you in?" → check service area immediately
   If out of area: decline gracefully, offer OLB link or referral
2. "And the street address?" → number + street name
3. "Is there an apartment or unit number?" → explicit ask, every time
4. Read back once to confirm, digit-by-digit for numbers
2. Always ask for unit/apartment explicitly

Missing apartment/unit numbers are the #1 cause of technician dispatch failures in home services. Customers almost never volunteer it unprompted. It must be a distinct, explicit question — not bundled into "full address including city and zip." Ask it every time for new customers.

Prompt copy
After collecting street address, always ask:
"Is there an apartment, unit, or suite number?"
Wait for a yes/no. If yes, collect it separately.
Do not skip this for residential addresses.
3. Read back digit-by-digit, once, then stop

Industry best practice for call centers: read numbers as individual digits, not as a whole number. "Four-five-one Oak Drive" not "four hundred fifty-one Oak Drive." This is how the customer said it; hearing it back the same way triggers instant recognition. Confirm once, move on — never repeat the address after confirmation.

Prompt copy
Read the address back once: "I have you at [number digit-by-digit]
[street name] — is that right?"

After confirmation: never say the address again in this call
except once in the final booking summary.
4. Validate silently — customer always wins disagreements

Fire the validation API silently after the customer confirms. If the validator agrees — great, continue. If the validator returns something different — do not surface it unless it's a major discrepancy (different street name entirely). Minor corrections (ZIP+4, suffix standardization) apply silently. If the customer disputes the result even once, accept their version and proceed. Customers know their own addresses better than any database.

Prompt copy
Validation is silent. If validator disagrees with customer:
- Minor difference (ZIP+4, St → Street): accept silently
- Major difference (wrong street/city): ask once: "Just to confirm —
  I'm seeing [validator result] — or is it [what you said]?"
- Customer says yours: accept theirs. Period. Do not re-ask.
- No match at all: accept what customer said, flag for review, proceed.
5. Pre-populate from caller ID for returning customers

When caller phone number matches a HCP customer record, load the address from the record and confirm in one turn: "Is this still for [address on file]?" This is both faster and dramatically more accurate than collecting from scratch. Avoca AI, ServiceAgent, and other home-services voice AI platforms all do this by default. It should be table stakes for CSRAI too — currently the AI re-collects even for known callers.

Prompt copy (Phase 2 — needs eng)
For known callers (phone match confident):
"Is this still for [address on file]?"
If yes → proceed, no collection needed.
If no → collect fresh using zip-first flow above.
Never ask a returning customer to spell out their address from scratch.
6. Consider switching to Smarty for US validation

USPS data has a known gap: 20M+ valid US addresses are missing from their database. These are predominantly new construction and rural addresses — exactly the customer base for HVAC, plumbing, and appliance repair. The current address validation failures in 18% of calls may be partly due to this gap. Smarty (formerly SmartyStreets) is USPS CASS-certified AND has a proprietary dataset of 20M+ non-USPS addresses. For home services, it's the strongest choice. USPS API is rate-limited to 60 requests/hour and should not be used in production.

Best validators for home services voice AI: Smarty (US residential, best coverage) · Google Address Validation API (global, Dialogflow-native) · Melissa Data (US + Canada, best for commercial addresses)
7. Name collection — ask for spelling once, then stop

For new customers whose name is not on file: ask for their name, then ask them to spell it. Confirm once. Never repeat their name again during the call — it sounds robotic and adds nothing. For returning customers, use the name in the greeting only.

Prompt copy
NAME RULE (new customers):
"Can I get your name?" → [customer responds]
"And can you spell that for me?" → [customer spells it]
"Got it, [Name]."

That's it. Do NOT repeat their name again during the call.
Do NOT say "Great, [Name]!" or "[Name], I have you down for..." mid-conversation.
The name is collected for the booking record only.

NAME RULE (returning customers):
Use the name on file in the greeting only: "Hi [Name], this is [org]. How can I help?"
Do NOT repeat it again after that.
Emergency call exception: For urgent calls (no heat, gas off, flooding), skip extended address collection and validation. Get ZIP code + street address only, confirm once, proceed immediately. Do not add apartment confirmation turns or run full validation on an emergency — speed is the priority. Full address reconciliation can happen post-dispatch.

Phase 1 — Prompt Only Biggest strategic change · No eng required

~2/3 of calls are existing customers. The AI needs to detect at call start whether the caller has an active appointment (→ Service Mode: help them) vs. no appointment (→ Sales Mode: book now). Currently the AI defaults to secretary mode regardless.

❌ Current
Collect name → look up record → ask reason for call → collect address → check services → check availability → if no slot, note for team
✅ v8
If active appt on file: "Hi [name], I can see you have [service] scheduled for [date] — how can I help?" → Service Mode

If no active appt: → Sales Mode: "The next opening I have is [day] at [time]. Can I book that for you?"
Prompt copy — paste directly
CALL MODE — determine at call start using pre-loaded caller context:

IF caller has an active or upcoming appointment:
  → SERVICE MODE
  Open with: "Hi [name], what can I help you with today?"
  Help them with their existing appointment (reschedule, cancel, confirm, add notes).
  Do NOT pitch a new booking unprompted.

IF caller has no active appointment OR is a new caller:
  → SALES MODE
  Your goal is to book an appointment.
  After identifying the service needed, immediately offer a specific time:
  "The next opening I have is [day] at [time] — can I book that for you?"
  Do NOT default to note-taking. Do NOT say "I'll have the team reach out" when a slot is available.

IF unclear: ask once — "Are you calling about an existing appointment or looking to schedule something new?"

📐 How We Should Be Counting This — Segmentation & Metrics

The current 1,804-session dataset treats all calls the same. That's wrong. A returning customer with an active appointment and a new customer calling to book for the first time are completely different use cases with different success criteria. Mixing them inflates or deflates every metric we track.

New Customers / No Active Appointment
→ Sales Mode — Primary Metric: Booked

These are the calls CSRAI should be optimized for. Success = confirmed booking. Every issue in this analysis (secretary mode, availability dead-end, tool call friction) is most damaging here — this is a new customer who may never call back.

Primary metric: call-to-booked rate
Secondary: time to booking confirmation, turns to book
Returning Customers with Active Appointment
→ Service Mode — Primary Metric: Resolved

These callers already have a booking. They're calling to reschedule, confirm, cancel, or ask a question. Counting them in "intent to book" inflates the conversion denominator — they were never trying to book in the first place. Success = resolved without escalation.

Primary metric: self-service resolution rate
Secondary: escalation to human rate, reschedule completion rate
⚠️ Current Measurement Gap

Right now we have no way to split the 1,804 calls into these two groups — the QA sheet has no caller_type field populated (all N/A), and we don't pre-load active appointment status before the call. Until we can tag calls as new vs. returning, our conversion rate, booking rate, and issue frequency stats are mixing two incompatible populations.

HCP AI — Context Architecture

Why context is the product · The case for a single context endpoint · Voice vs. SMS tradeoffs
The quality of an AI interaction is almost entirely determined by what context it has when it starts talking. The research is unambiguous on this. Right now HCP AI fetches context in small pieces, mid-call, via tool calls. That approach has a measurable latency cost, a documented accuracy cost, and a compounding effect on call outcomes. This document makes the case for a different architecture — one rich context endpoint, loaded once at session start, consumed by every channel.
🧠 Why Context Is the Product

The LLM itself is not the differentiator. Every competitor has access to the same models. The differentiator is what you put in the context window before the model responds. The research on this is now extensive and consistent — all of it from the second half of 2025 and into 2026.

Context window size doesn't solve quality — structure does (Epoch AI, 2025)

Even as context window sizes have grown dramatically, accuracy degrades with scale. GPT-4 accuracy drops from 96.6 to 81.2 as context grows from 4K to 128K tokens — the model struggles to use what it can technically hold. Bigger windows don't fix the problem. Structured, well-ordered context does. The implication for HCP AI: don't dump everything in — structure what goes in and sequence it to put the most relevant information first.

⏱️ The Latency Problem — Why Tool Calls Break Voice AI

In a voice conversation, latency isn't a technical metric — it's directly perceived by the caller as hesitation, confusion, or incompetence. Every mid-call tool call adds a full additional inference cycle. In voice, that's the difference between a natural conversation and a broken one.

Two inference passes vs. one. A single mid-call tool call roughly doubles LLM latency for that turn. HCP AI currently makes multiple tool calls per call to look up Pro info, availability, and customer data. Each one adds ~500ms to the caller's wait. This is not a prompt engineering problem — it's an architecture problem.
The token-latency tradeoff

Pre-loading context isn't free either. Every additional 500 tokens adds roughly 100ms to first-token latency. A well-structured 5,000-token context block adds roughly 1,000ms — but this is a one-time cost paid at session start, before the AI speaks, not a recurring cost paid mid-conversation on every turn that needs data. The Anthropic Nov 2025 research makes this tradeoff concrete: replacing tool search with pre-loaded structured context reduced total tokens consumed from 77K to 8.7K — a structural win, not just a latency one. Pay the context cost once at the top, or pay it repeatedly throughout the call.

📡 Voice vs. SMS — Different Latency Budgets, Same Context Need

The context we need is the same across channels. The latency constraints are not. This shapes how and when we can do richer processing.

📞 Voice (HCP AI — Today)

Latency budget: <500ms end-to-end
  • Human conversation gap is 200–300ms — that's the bar
  • Context must be fully pre-loaded before the AI speaks its first word
  • Zero tolerance for mid-call tool calls on data that can be known upfront
  • >1,000ms response = 40% more abandonment (Retell AI, Jul 2025)
  • Production median today is 1,400–1,700ms — 2× over target (AssemblyAI, Dec 2025)
  • Context must be cached and ready — not assembled on demand
  • Richer processing (e.g., job history distillation) must happen offline, pre-session

💬 SMS (HCP AI — Coming)

Latency budget: 1–3 seconds acceptable
  • No natural conversational rhythm to match — async by default
  • Callers expect slight delays in text; <3s feels instant
  • More room for richer real-time context assembly at message receipt
  • Can do live lookups that would be too slow for voice
  • Can fetch full job history inline, not just cached summaries
  • Still benefits from pre-loaded Pro + Customer base context
  • Opens door to richer agentic actions (booking, estimate sending, follow-up)
Same endpoint, different payload depth. The context endpoint can serve both channels. For voice: return the pre-cached, compressed payload — built for latency. For SMS: return the full payload including richer data that would be too heavy for voice. A single channel parameter controls which profile the endpoint returns.
🔌 The Solution — One Endpoint, Rich Structured JSON

We need a single endpoint that any HCP AI channel can hit at session start and get back a complete, structured context block. No mid-session tool calls for data that can be known in advance. The endpoint accepts what's known at that moment — phone number, customer ID if resolved, Pro ID always — and returns everything we have.

// The HCP AI Context Endpoint GET /v1/ai/context — Parameters (pass what you have) — pro_id string // required — always known from the HCP AI phone number phone_number string // optional — caller ANI; used to resolve customer_id customer_id string // optional — if known (from prior resolution) channel enum // "voice" | "sms" — controls payload depth — Response — pro // Pro context block (see Pro Context spec) business // identity, contact, licenses service_area // active zip codes, cities services // what they do, what they don't hours // availability, timezone, after-hours rule calendar // next available slots team // technician names and roles ai_config // greeting, escalation number, custom instructions intelligence // pre-built from daily batch agent (job history distillation) customer // Customer context block — null if phone not matched identity // name, phones, addresses, tags, notes jobs // all jobs — status, type, appointments estimates // open estimates call_history // prior HCP AI transcripts + outcomes sms_threads // recent messages campaigns // marketing sent, opened, clicked memberships // active recurring plans meta customer_resolved // true | false channel // "voice" | "sms" context_age_ms // how fresh the pro.intelligence block is
The AI has one job at session start: hit this endpoint, receive the block, inject it into the system prompt. No decisions, no multiple calls, no sequential lookups. One call, full context. The endpoint handles the complexity of assembly — the AI just consumes the result.
🏗️ How the Context Is Built — Three Layers

Not all context has the same freshness requirement or assembly cost. We split it into three layers with different update cadences — the endpoint assembles and serves all three at call time.

⚡ Real-Time

Fetched at session start

Volatile data that changes frequently. Assembled fresh for every call. Must be fast — cached at the field level where possible.

  • Available calendar slots
  • Open jobs / active appointments
  • Open SMS threads
  • Technician availability today
  • Customer phone → identity lookup
🕐 Daily Batch

Pre-built, stored, retrieved

Rich intelligence that takes time to generate. Built once per day per Pro by a background agent. Retrieved instantly at call time — zero assembly latency.

  • Job history distillation (180 days)
  • Service frequency patterns
  • Real pricing ranges from invoices
  • Tech note themes
  • Pro narrative summary paragraph
📋 Near-Static

Cached, refreshed on change

Configuration data that changes rarely. Cached aggressively. Invalidated and refreshed when a Pro updates their settings.

  • Business name, hours, timezone
  • Service area (configured)
  • AI config (greeting, escalation)
  • Team roster
  • OLB settings
Why the daily batch layer matters: 180 days of job history is too large to process at call time — hundreds to thousands of jobs per Pro. The daily batch agent runs this processing offline, produces a compact structured profile (~2–3k tokens), and stores it for instant retrieval. This is how we get the richest possible Pro intelligence with zero latency cost on live calls.

❓ Open Questions
  • 1What's the target latency for the context endpoint to return, end-to-end, for the voice channel? This defines the SLA for every sub-component — real-time fetches, cache reads, identity resolution.
  • 2For voice: do we inject the full context block into the system prompt on every session, or only fields that are relevant to the detected caller intent? A shorter, focused prompt has lower TTFT (~100ms per 500 tokens) — worth defining the tradeoff.
  • 3For SMS: the relaxed latency budget opens the door to real-time job history fetching rather than relying on the daily batch cache. Engineering to define what additional data is worth fetching live in SMS vs. voice.
  • 4How does the endpoint handle a customer_id that's known mid-conversation (spouse on different phone)? This is the get_customer_context(customer_id) tool call — a deliberate exception to the pre-load model. Documented in the Customer Context spec.
  • 5What's the caching strategy for Pro near-static data? Cache invalidation on Pro settings change is straightforward — the bigger question is how to handle partial invalidation (hours change but services don't).

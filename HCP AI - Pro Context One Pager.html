HCP AI — Pro Context Service

Engineering Requirement · Chris Kosednar → Montrell Williams · 2026-03-17
A shared backend service that pre-loads everything the AI needs to know about the Pro's business before it handles any interaction. The AI should never need to make a mid-call tool call to find out what services a Pro offers, what their hours are, or whether they cover a zip code. All of that lives here, injected at session start.
📞 Voice (HCP AI) — inbound calls today
💬 SMS (HCP AI) — inbound texts, coming soon
⚡ Future channels — any AI interaction acting on behalf of this Pro
Unlike customer context — which requires identity resolution — Pro context is always known. The org_id is determined at session start (the Pro's HCP AI phone number maps to an org). There is no ambiguity. Load it once, keep it current.

Goal: Zero mid-interaction tool calls for Pro info. The AI should already know what services this Pro offers, what their service area is, and when they're available. Dead air from mid-call lookups for basic Pro details is the problem this solves.

❓ Open Questions for Engineering
  • 1What's the realistic latency to assemble the full Pro context payload at session start? Which fields require joins vs. are cached? Target: assembled before first AI utterance.
  • 2How stale can Pro context be? Business hours and services change infrequently — cache is probably fine. Calendar availability changes minute-to-minute — needs to be live or near-live. Engineering to define TTL per object type.
  • 3Available slots: do we inject the next N slots into the system prompt, or does the AI query availability dynamically? Full pre-load vs. tool call for scheduling is a key architectural decision.
  • 4Pricebook can be very large for some Pros (hundreds of line items). What's the right truncation strategy — top N services by booking frequency? Services matching detected caller intent?
  • 5Custom instructions is a free-text field. Do we inject it verbatim, or does it get parsed/validated before it reaches the AI? Risk of prompt injection from Pro-entered content.
  • 6How does Pro context interact with Customer context when both are loaded? Define injection order and any field collision rules (e.g., Pro says "don't quote prices" but customer_id lookup has past invoice amounts).
